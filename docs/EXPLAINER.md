# self-improve-arch 解説マニュアル（なにこれ / なぜ / しくみ）

予備知識ゼロで読めるように書いています。使い方の手順は [USAGE.md](USAGE.md) を見てください。

---

## 1. 一言でいうと

> **「このコード改善案、本当に採用していい？」を、感覚や AI の自己申告でなく、
> 実テスト＋実ベンチ＋統計＋レビューで厳密に判定する“採否の決定ツール”です。**

入力は「ある関数の、複数の書き換え候補」。出力は「どれを採用すべきか（あるいは全部不採用か）」を
**証拠つき**で返します。AI が自律的にコードを書き換え続ける装置ではありません（→ §6）。

---

## 2. なぜ要るのか（何を解く道具なのか）

「速くしたつもり」「直したつもり」は、いろんな形で**だまされ**ます。素朴にやると:

| 素朴なやり方 | だまされ方 |
|---|---|
| AI/人が「速くなった」と言うのを信じる | 自己申告は検証されていない |
| 1回計って速ければ採用 | 計測ノイズで偶然速く見えただけ |
| 候補を10個試して一番速いのを採用 | 10回くじを引けば1個はまぐれで「有意」になる（多重比較） |
| その一番速い候補をそのまま採用 | 「最良を選んだ」こと自体が効果を過大評価する（winner's curse） |
| 同じテストデータで何度も確かめる | データに過学習して「効いてるつもり」になる（holdout 枯渇） |
| テストが緑なら OK | テスト範囲外の欠陥（共有状態・並行バグ等）は見逃す |
| 「計測コードがあるから安心」 | 候補が計測そのものを偽装できる（計測捏造） |

**self-improve-arch は、この“だまされ方”を一つずつ潰す層を重ねたものです。**
各層は「素朴にやると○○でだまされる → だからこの装置」という対応で理解できます。

## 3. 全体の流れ（1 サイクル）

```
あなたの候補ファイル群
        │
        ▼
   slate（K 個の候補をまとめて評価）
        │
        ├─ 各候補について:
        │     ① sandbox … 隔離コピーで「実テスト」+「実ベンチ」を実測
        │     ② Reviewer 2体 + Judge … コード可視の欠陥を veto（必要条件）
        │     ③ gate … テスト合格 AND 主要KPI が有意改善 AND ガードレール非回帰
        │              （K 個同時比較なので Bonferroni で α/K に補正）
        │
        ├─ valid な中から「最良の 1 つ」を選ぶ（1 サイクル最大 1 採用）
        │
        ├─ ④ fresh confirm … 選んだ最良を“探索に未使用の別データ”で再確認
        │     （⑤ query-budget … 確認用データには使用回数の上限。枯渇したら停止）
        │
        ▼
   確認も通ったものだけ「採用」。全候補が KB(sqlite) に証拠とともに記録される。
```

## 4. 各装置を「どんな“だまされ方”を防ぐか」で

### ① 床 = 実テスト + 実ベンチ（自己申告を信じない）
採否の土台は **2 つの実測**だけ:
- **実テスト**: 候補を隔離コピーに適用し `pytest` を実行。pass/fail が採否入力。
- **実ベンチ**: baseline（元の実装）と候補を**同一プロセスで交互に計測**し、速度差を統計（z 検定）で見る。

候補が「速いです」と言っても無視。**測った値だけ**が床です。

### ② Reviewer 2体 + Judge（テスト緑でも危険なら止める・ただし床は動かさない）
テストは「やってほしいこと」しか見ません。テスト範囲外の欠陥（例: 引数に共有 mutable default を
持ち並行で壊れる／公開シグネチャを勝手に広げる）はコードを**見れば**分かります。
- **Reviewer** は観点別（safety / scope）に**コード可視の欠陥**を `blocking` で指摘。
- **Judge** は「1 人でも blocking があれば不承認」という**決定的ルール**で集約（LLM の気分ではない）。

重要: この層は **veto（拒否）しかできません**。テスト・KPI の床は動かせず、「採用の必要条件が
1 つ増える」だけ。ゲート不合格を救済することは構造上できません。

### ③ Bonferroni 補正（候補を並べると偽陽性が膨らむ）
K 個の候補を同じ baseline と比べて「一番」を採るのは、**K 回の同時検定**です。無補正だと
「どれかがまぐれで有意」になる確率が膨らみます（K=5・α=0.05 で約 23%）。
そこで各候補を **α/K** の厳しさで検定します（family 全体の誤採用を α 以下に抑える）。

### ④ fresh confirm slice（“最良を選んだ”こと自体のバイアスを外す）
たとえ Bonferroni で「まぐれ採用」を抑えても、**最良を argmax で選ぶと、その効果量は楽観的に
偏ります**（winner's curse）。そこで、選ばれた 1 つを **探索に一度も使っていない別データ**で
**もう一度だけ**検定し、**再現したものだけ採用**します（search ⊥ confirm）。
探索ノイズで勝っただけの候補は、ここで効果が消えて落ちます。

### ⑤ query-budget（確認用データは無限ではない）
confirm を毎回同じデータで叩き続けると、そのデータにも過学習します（longitudinal な枯渇）。
そこで confirm 用データに「使える回数の上限」を持たせ、**消費を KB に永続記録**。使い切ったら
次の新しいデータへ。全部使い切ったら **採用を止めます**。
ねらいは情報を増やすことではなく、**「黙って過学習する」を「枯渇で止まる」に変える**こと。

### ⑥ 計測整合性の防御（計測捏造 — ただし“境界”ではない・重要）
候補は untrusted なコードとして実行されます。素朴な実装だと、候補が `time.perf_counter` や
結果の直列化を差し替えて **latency を捏造**できます（実改善ゼロでも「採用」に到達させられる）。
- 防御として、計測器を候補ロード前に**束縛退避**し、ベンチ出力長を照合します。
- **ただしこれは“境界”ではありません。** Python では reflective に任意コードへ到達でき、
  `os._exit` や gc 操作で結局 forge できます（実際にデモで確認済み）。
- **OS 分離は `--isolation` で実装しました**（既定 rlimit=非特権の DoS 床、`--isolation docker`
  で `--network none`＝候補の network exfiltration / DoS / FS 逸脱を断つ）。ただし**計測値そのものの
  捏造**（sandbox 内で候補が偽の timing を返す）は OS 分離では塞げない別問題で、候補が触れない
  外部計測が要る（残る天井・N 照合は部分緩和）。

→ 運用の鉄則は「**信頼できない候補は `--isolation docker`（不可なら外側のコンテナ/OS 分離）で
動かす**」です（[USAGE.md](USAGE.md) 安全境界）。

## 5. 出てくる言葉（用語ミニ辞典）

- **候補 (candidate)**: 改善対象モジュール（例 `dedupe.py`）の**全文を差し替える**ソース 1 個。
- **slate**: 1 サイクルでまとめて評価する候補の束（K 個）。
- **baseline**: 現行の実装（候補の比較対象）。
- **gate（ゲート）**: 採否を決める決定的ロジック（テスト・KPI 有意性・ガードレール）。
- **ADOPT / REJECT / NOT_ADOPTED**: 採用 / テスト不合格で却下 / テストは通るが基準未達。
- **KPI / primary**: 主要指標（既定は latency = 低いほど良い）。
- **KB**: sqlite の知識ベース。全候補の採否・理由・予算消費を記録。

## 6. 正直な限界（このツールが“しないこと”）

このプロジェクトの結論そのものが、**「内部測定には天井がある」**です。

- 内部の測定・検査で縛れるのは **fast / gross / in-band**（速い・粗い・観測済み次元の）失敗だけ。
- **slow / subtle / construct-drift**（じわじわ・巧妙・KPI が現実とズレる）失敗、
  そして**計測捏造の真の防御**、**本当に新しいデータ**は、**内部では原理的に得られません**。
  これらは外部（別系統のモデル・人間の判断・第三者監査・OS 分離・新規データ供給）でしか床になりません。
- よって本ツールの正しい立ち位置は **「operator が用意した候補を厳密に採否する“決定の床”」**であって、
  **AI が自律的に最適化し続ける装置ではありません**。実 LLM builder/reviewer は配線だけ存在し、
  この workflow では自動起動しません（`claude -p` 不使用・claude-in-claude 回避）。

各機能は「装置を足す → 残る天井を明記する → 次の装置でその天井を縛る」という積み重ねで、
最後の天井（計測捏造の境界・データ枯渇）は**外部に委ねる**と正直に書いてあります。
それが `test_security.py` などにテストとして固定されています。

→ 「**これで自律的な再帰的自己改善（RSI）は実現できるのか？**」への踏み込んだ答えは
[RSI-LIMITS.md](RSI-LIMITS.md)（結論: いいえ＝実現でなく封じ込め）。
